# Independent Forward Test Playbook

这个文件描述 `impl-phase-coach` 下一步应该怎么做：在一个尽量干净的新会话里，独立跑一轮真实试跑，并且把结果记录成可纠偏的材料。

## Discovery Prerequisite

先确认这件事：当前这个 skill 位于 `D:\Desktop\impl-phase-coach`，它是一个有效的本地 skill 包，但默认**不会**被 Codex 新窗口自动发现。

原因：

1. 当前 `CODEX_HOME` 未设置。
2. 默认自动发现目录是 `C:\Users\17188\.codex\skills`。
3. 当前 skill 不在这个自动发现目录里，而是在工作目录中。

所以独立试跑前，你必须二选一：

### Option A: 安装到自动发现目录

把整个 skill 目录复制或移动到：

`C:\Users\17188\.codex\skills\impl-phase-coach`

这样新窗口里可以直接按名字触发它。

### Option B: 在新窗口里显式写路径

不要假设新窗口已经认识这个 skill。直接在请求里写：

`Use $impl-phase-coach at D:\Desktop\impl-phase-coach to ...`

这更适合当前还在本地工作目录迭代、但暂时不想正式安装的时候。

如果你只是想先做一次独立试跑，优先推荐 `Option B`，因为它改动最小。

## Goal

验证这个 skill 在脱离当前上下文后，是否还能稳定做到：

1. 先识别当前实现阶段。
2. 先输出阶段目标、产物、退出条件。
3. 给出知识权重，并在边界容易混淆时解释当前最相关的边界点。
4. 在代码阶段提供 A/B 两种推进方式。
5. 保持实现期教练风格，而不是退回普通助理模式。

## How To Run

按下面顺序执行：

1. 新开一个尽量干净的会话。
2. 先确认这次试跑是否已经满足 `Discovery Prerequisite`。
3. 不要粘贴这次对话历史，只给 skill 名称和测试请求。
4. 第一轮只测一个小请求，不要一上来测复杂项目。
5. 把首轮回复原样保存下来，不要先人工润色。
6. 按本文件的记录模板和判定标准打标签。

## Recommended Test Requests

先用下面 3 个请求之一，不要三个一起测：

### Request A: 阶段识别

“用 $impl-phase-coach 带我按阶段实现一个小功能。先判断我当前处于哪个实现阶段，给出当前阶段目标、产物和退出条件，不要直接写完整实现。”

适合验证：

- 是否会先识别阶段
- 是否会按默认输出协议开头
- 是否会避免整包实现

### Request B: 状态语义

“我现在最乱的是 `DENIED` 和 `FAILED` 的区别。你按实现阶段带我推进，先不要展开高层战略。”

适合验证：

- 是否会判到状态层相关阶段
- 是否会避免回到完整方法论分析
- 是否会把问题压缩在当前最小闭环

### Request C: 纠偏场景

“我已经写了一半代码，你别全补，先按文件边界、对象字段、状态语义和测试矩阵帮我判断我当前处于哪个阶段，然后给我关键位置示例代码。”

适合验证：

- 是否会进入教练式纠偏
- 是否会保留 A/B 推进方式
- 是否会优先检查正确的纠偏维度

## Recording Template

每次试跑至少记录下面这些字段：

1. `测试请求`
2. `skill 是否真正被发现并加载`
3. `首轮回复是否先识别当前阶段`
4. `是否给出阶段目标 / 产物 / 退出条件`
5. `边界混淆时是否解释了当前边界点`
6. `是否给出知识权重`
7. `是否提前跨阶段展开`
8. `是否直接给完整实现`
9. `如果进入代码阶段，是否提供 A/B 选项`
10. `风格是否保持工程化、结构化、教练式`
11. `最终判定：通过 / 可恢复失败 / 失败 / 前置条件失败`
12. `失败类型：discovery failure / 阶段误判 / 协议缺失 / 风格退化 / 其他`

## Pass Criteria

满足下面条件，可判为 `通过`：

1. skill 已被真正发现并加载。
2. 首轮回复先识别当前阶段。
3. 首轮回复明确包含阶段目标、产物、退出条件。
4. 首轮回复在需要时能解释当前边界点。
5. 回答没有提前跨到后续阶段。
6. 如果进入代码阶段，明确给出 A/B 选项。
7. 风格没有退回普通助理或高层战略模式。

## Precondition Failure Criteria

满足下面任一情况，判为 `前置条件失败 / discovery failure`：

1. 系统明确说 `$impl-phase-coach` 在当前环境不存在。
2. skill 没被成功加载，就直接 fallback 到别的 skill 或普通回答。
3. 新会话没有满足 `Discovery Prerequisite`，就开始评测输出质量。

修法：

- 先修发现路径，再重新跑同一条请求。
- 不要把 fallback 回答当作这个 skill 的输出样本。

## Recoverable Failure Criteria

满足下面任一情况，判为 `可恢复失败`：

1. 阶段判断基本对，但结构不完整。
2. 漏了知识权重或 A/B 选项。
3. 首轮回复没有按最小骨架展开，但核心方向基本正确。
4. 风格轻微漂移，但仍能看出是在按阶段推进。

修法：

- 只补缺失的协议项或结构项，不要重写整个 skill。

## Hard Failure Criteria

满足下面任一情况，判为 `失败`：

1. skill 已被加载，但没有识别当前阶段就直接写代码。
2. skill 已被加载，但直接给完整实现。
3. skill 已被加载，但明显回到路线图、战略讨论或泛化方法论分析。
4. skill 已被加载，但在纠偏场景里先做表层风格建议，而不是先检查阶段成立条件。

修法：

- 先定位失败类型，再回到对应协议位置做最小修复。
- 不要一失败就全面改写所有文档。

## Next Action After Test

如果结果是 `通过`：

1. 继续做第二轮更具体场景试跑。
2. 优先测状态语义场景或纠偏场景。

如果结果是 `前置条件失败`：

1. 先修 discovery 路径问题。
2. 用同一条请求重新跑，不要先换请求。

如果结果是 `可恢复失败`：

1. 只修最小缺陷。
2. 修完后重新跑同一个请求，确认问题是否消失。

如果结果是 `失败`：

1. 先回到失败类型定位。
2. 只修根因所在的协议段落。
3. 修完后重新跑第一轮请求，不要立刻换更复杂请求。
