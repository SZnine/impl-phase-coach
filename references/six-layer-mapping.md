# Six-Layer Mapping

这个文件是 `impl-phase-coach` 的 6 层边界说明，用来在边界容易混淆时帮助解释当前为什么先处理这一边。

## 1. 模块层

- 目标：决定文件边界、职责归属、模块不串味。
- 提问方式：这段代码应该放在哪个文件，属于哪个模块，为什么不放到别的模块。
- 常见错法：为了图快把多个职责揉进同一个文件，导致后续边界失真。

## 2. 接口层

- 目标：决定核心对象、输入输出契约、返回结构。
- 提问方式：函数签名是否稳定，对象字段是否清晰，返回值是否表达单一语义。
- 常见错法：把临时字段塞进核心对象，或让返回结构同时承担状态和值。

## 3. 状态层

- 目标：决定主状态流、`DENIED` vs `FAILED`、状态和值域。
- 提问方式：这是业务拒绝、执行失败、还是正常空结果；状态是否有稳定枚举和值域。
- 常见错法：把状态、异常、返回值混在一起，导致上层无法可靠分支。

## 4. 最小实现层

- 目标：决定当前阶段最小闭环、先做什么不做什么。
- 提问方式：这一阶段最小成立版本是什么；哪些需求刻意不做，留到下一阶段。
- 常见错法：阶段边界没冻结，顺手把后续阶段也一起实现掉。

## 5. 运行观测层

- 目标：决定 `request_id`、事件链、错误码、审计字段。
- 提问方式：当前实现是否能被追踪；关键状态迁移是否可观测；失败后能否定位。
- 常见错法：只写业务逻辑，不留下可追踪字段，后续排错只能靠猜。

默认检查点：

1. 是否存在稳定的 `request_id` 或等价关联字段。
2. 是否能看见关键事件链，例如“收到请求 -> 参数校验 -> 进入执行 -> 返回结果”。
3. 是否区分业务拒绝、执行失败和系统异常。
4. 是否存在可复用的错误码或错误分类，而不是只返回一段字符串。
5. 是否保留最小必要的审计字段，便于后续复盘。

## 6. 验收层

- 目标：决定测试矩阵、退出条件、版本成立标准。
- 提问方式：当前阶段测试覆盖了哪些分支；退出条件是什么；什么算这一版成立。
- 常见错法：测试只验证 happy path，没有和当前阶段退出条件对齐。

默认检查点：

1. 是否至少覆盖正常路径、拒绝路径、失败路径三类分支。
2. 测试是否直接对齐当前阶段退出条件，而不是泛泛测一遍。
3. 是否区分“当前阶段成立”与“最终形态成立”。
4. 是否明确哪些问题属于必改项，哪些可以在下一阶段保留。
5. 是否能用测试、日志或明确检查步骤支撑“本阶段可冻结”的判断。

## Default Review Checklist

每次阶段化实现或纠偏时，优先按下面顺序检查：

1. 模块边界有没有串味
2. 接口对象和字段有没有漂移
3. 状态语义有没有混淆
4. 最小闭环有没有被提前扩写
5. 运行观测字段是否足以定位问题
6. 验收测试是否对齐退出条件

## Usage Rule

- 这 6 层是边界透镜，不是每轮必报清单。
- 只有当边界容易混淆时，才借助它们解释区别。
- 更重要的是说明“为什么当前先处理这一边，而不是另一边”。
