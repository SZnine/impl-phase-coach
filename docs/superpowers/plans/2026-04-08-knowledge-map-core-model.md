# Knowledge Map Core Model Draft

> 当前文档不是实现计划，而是知识地图工作台的核心模型草案。它的职责是冻结产品核心对象、关系、视图与 LLM 治理边界，为后续实现提供稳定输入。

## 1. 当前阶段定位

### 当前阶段目标
1. 把“知识地图到底是什么”从口头讨论压成正式草案。
2. 冻结第一版核心对象、关系、视图与治理边界。
3. 避免后续实现再次把知识图做成：
   - 纯展示页
   - 纯项目挂载图
   - 纯抽象百科

### 当前阶段产物
1. 一套知识地图核心对象模型。
2. 一套第一版关系类型与证据呈现规则。
3. 一套 LLM 自动维护与 proposal 治理边界。

### 当前阶段退出条件
1. 后续实现可以直接引用本草案，而不需要重新高层收敛。
2. “节点是什么、关系是什么、证据怎么挂、焦点怎么组织、LLM 能做什么”这五个问题有明确答案。

---

## 2. 产品定义

这不是一个简单的知识图页面，而是：

`一个以知识地图为核心索引层、由稳定知识节点、本地用户画像状态、项目事实证据和 LLM 维护关系共同组成的长期个人工作台。`

它的目标不是生成一张静态图，而是：
1. 把项目、题目、回答、错题、proposal 等事实留痕。
2. 把这些事实不断抽象成稳定知识节点与关系。
3. 让用户在长期使用中获得一张“可回溯、可生长、可治理”的个人知识地图。

---

## 3. 核心原则

### 3.1 事实静态留痕，抽象动态重组
1. 项目、题目、回答、评估、错题、proposal、execution 是事实层。
2. 知识节点、关系、焦点簇、解释、去重建议属于抽象层或组织层。
3. 事实层不能被 LLM 静默改写。
4. 抽象层允许被 LLM 动态维护，但关键结构变化必须可审计、可治理。

### 3.2 稳定骨架产品化，动态抽象 LLM 化
必须产品化的部分：
1. 节点本体
2. 关系本体
3. 证据锚点
4. 用户状态
5. 焦点簇
6. proposal/action/execution

适合 LLM 处理的部分：
1. 新节点归类
2. 关系候选
3. 短解释生成
4. 去重建议
5. 重构建议
6. 视角整理

### 3.3 地图本体优先，证据按需展开
1. 默认主图只展示知识节点与知识关系。
2. 项目、错题、题目、回答、proposal 不是默认主图节点。
3. 证据锚点以节点附属证据层形式存在，按需展开。

---

## 4. 分层结构

### 4.1 四层抽象结构

#### L0 证据层
用于挂载：
1. 项目
2. 阶段
3. 题库
4. 题目
5. 回答
6. 评估
7. 错题
8. proposal / execution

职责：
1. 提供来源
2. 提供可回溯锚点
3. 证明节点或关系为何成立

#### L1 局部理解层
表示当前项目或阶段里直接暴露出的局部理解点。

例如：
1. 状态与返回值混淆
2. proposal status 和 execution status 边界不稳
3. session 与 durable facts 边界混淆

#### L2 稳定知识层
表示相对稳定、可跨项目复用、可长期保留的知识节点。

例如：
1. 状态与返回值分离
2. 事实层与抽象层分离
3. 状态机
4. 索引
5. 模块边界

#### L3 元原则层
表示更高层的方法论、组织原则、治理原则。

例如：
1. 先稳事实层，再做抽象增强
2. 底层事实静态留痕，上层知识动态重组
3. LLM 可以生长地图，但不能静默改写地图骨架

### 4.2 显示策略
1. L3 默认弱显式，不常驻主视图。
2. 当用户切到“原则/方法视角”时，L3 才强化出现。

## 5. 节点模型

### 5.1 节点类型
第一版节点类型最小集合：
1. `foundation`
2. `concept`
3. `decision`
4. `mistake`
5. `method`
6. `question`

说明：
1. `foundation` 是一等节点，不能隐含在 `concept` 里。
2. 这保证知识图不会只剩抽象方法论，也能承载可复习、可迁移、可共识化的基础知识。

### 5.2 scope
建议最少包含：
1. `universal`
2. `project_bound`
3. `personal`

作用：
1. 区分通用基础知识、项目绑定知识和个人薄弱/个性化知识。

### 5.3 进入机制
节点进入系统采用三段式：
1. `Seed`
2. `Activation`
3. `Evolution`

#### Seed
预置一部分基础知识底盘。

#### Activation
当项目、题目、回答、错题命中这些节点时，为节点挂上证据与用户关系。

#### Evolution
LLM 根据新增事实补关系、补分类、补解释、补重构建议。

## 6. KnowledgeNode 与 UserNodeState

### 6.1 核心判断
节点本体应稳定，掌握状态应个性化。

因此必须显式拆成两层：
1. `KnowledgeNode`
2. `UserNodeState`

### 6.2 KnowledgeNode
建议最少字段：
1. `node_id`
2. `node_type`
3. `label`
4. `abstract_level`
5. `scope`
6. `canonical_summary`
7. `seed_or_generated`
8. `source_refs`
9. `merge_candidate_meta`

它回答的是：
1. 这是什么知识对象
2. 它属于什么类型
3. 它在地图里的稳定位置是什么

### 6.3 UserNodeState
建议最少字段：
1. `node_id`
2. `activation_status`
3. `mastery_status`
4. `review_needed`
5. `weak_signal_count`
6. `last_seen_at`
7. `linked_project_count`
8. `confidence`

它回答的是：
1. 这个用户当前与该节点是什么关系
2. 这个节点是否需要回顾
3. 这个节点当前是不是薄弱点

### 6.4 冻结规则
1. 节点本体不直接承载高频变化的掌握状态。
2. 用户画像状态不反过来改写知识本体定义。

---

## 7. 关系模型

### 7.1 第一版公开关系最小集合
1. `supports`
2. `depends_on`
3. `abstracts`
4. `causes_mistake`
5. `evidenced_by`

### 7.2 各关系语义

#### supports
A 支撑 B 的理解或成立。

#### depends_on
A 依赖 B，没有 B，A 难以成立。

#### abstracts
A 是 B 的抽象总结，或 B 是 A 的具体实例。

#### causes_mistake
A 的缺失或不稳导致某种错误、误区或薄弱表现。

#### evidenced_by
某节点或某关系可被项目事实证明。

### 7.3 第一版不直接公开的内部维护辅助关系
1. `duplicate_candidate`
2. `possible_merge`
3. `possible_reparent`

说明：
1. 这些关系服务网络治理，不应与公开关系混成一层。

---

## 8. 证据锚点

### 8.1 核心判断
证据锚点不是地图本体，但必须可回溯。

### 8.2 默认呈现规则
1. 项目、错题、题目、回答、proposal 不默认作为主图节点展开。
2. 默认主图中只显示：
   - 知识节点
   - 知识关系
3. 每个知识节点附带轻量证据摘要：
   - 关联项目数
   - 关联错题数
   - 关联题目数
   - 关联回答数
   - 最近一次命中来源

### 8.3 展开策略
1. 选中节点后，再展开证据面板。
2. 只有在“证据视角”下，证据锚点才允许进入图中作为次级节点显式展示。

---

## 9. FocusCluster

### 9.1 核心判断
默认主视图不围绕单个节点，而围绕一个知识片区展开。

因此第一版需要显式对象：
`FocusCluster`

### 9.2 对象性质
1. 它不是长期稳定知识本体。
2. 它是可缓存、可重算、可解释、可 pin 的视图级组织对象。

### 9.3 第一版对象字段建议
1. `cluster_id`
2. `title`
3. `center_node_id`
4. `neighbor_node_ids`
5. `focus_reason_codes`
6. `focus_reason_summary`
7. `generated_from`
8. `confidence`
9. `last_generated_at`
10. `is_pinned`
11. `status`

### 9.4 两层结构判断
长期应存在两层：
1. 全局候选焦点簇
2. 用户焦点簇

但第一版只落：
`用户侧 FocusCluster`

---

## 10. 默认地图入口与主视图

### 10.1 入口规则
用户不直接裸进全图，而是：

`先进入地图摘要页，再进入地图主视图`

### 10.2 地图摘要页第一屏默认三块
1. `当前焦点簇`
2. `当前薄弱热点`
3. `基础知识热点`

### 10.3 焦点簇卡片最小字段
1. 焦点簇标题
2. 中心节点
3. 2~3 个关键邻居
4. 为什么它现在重要
5. 进入地图按钮

### 10.4 主视图组织方式
默认主视图不是：
1. 按抽象层直接展开
2. 按节点类型直接展开
3. 按项目挂载直接展开

默认主视图应是：

`工作地图视图`

其中心选择原则是：
`当前相关 × 当前薄弱 × 长期重要`

### 10.5 焦点选择来源优先级
1. `user_pinned_focus`
2. `system_current_focus`
3. `system_growth_focus`

### 10.6 主视图默认展开密度
冻结为：

`1 跳强关系 + 少量 2 跳精选关系`

说明：
1. 不默认展开全量二跳。
2. 不默认展开大量 `evidenced_by` 边。

---

## 11. 节点详情卡片

### 11.1 最小结构
节点详情卡片默认包含 4 个区：
1. 本体区
2. 用户状态区
3. 关系区
4. 证据摘要区

### 11.2 默认重心
冻结为：

`知识解释优先，行动建议次之`

说明：
1. 详情卡首先服务知识理解，不首先服务任务按钮。
2. 行动建议存在，但不应喧宾夺主。

### 11.3 用户状态呈现
冻结为：

`默认轻染色 + 可切换强化图层`

说明：
1. 默认态保留地图结构。
2. 强化图层才把状态当主视角放大。

---

## 12. focus_reason_codes

第一版最小集合建议为：
1. `current_project_hit`
2. `weak_signal_active`
3. `foundation_hot`
4. `cross_project_reuse`
5. `recently_changed`
6. `high_structural_importance`

### 12.1 使用规则
1. 每个焦点簇默认最多绑定 2~3 个 code。
2. reason code 用于解释“为什么现在重要”，不用于表达：
   - 节点类型
   - proposal 状态
   - LLM 来源
   - 证据明细

---

## 13. 低成本模型输入边界

### 13.1 最小输入三层
1. 节点静态本体
2. 当前静态信号
3. 允许引用的证据摘要

### 13.2 禁止越界生成
低成本模型绝不能自己发明：
1. 新原因
2. 新关系
3. 新证据
4. 新定义
5. 超出模板骨架的长解释

### 13.3 使用定位
低成本模型只适合：
1. 短解释生成
2. 轻摘要改写
3. 候选标签补充

不适合直接做：
1. 节点合并
2. 节点重命名
3. 层级升降
4. 主关系重构

---

## 14. LLM 权限边界

### 14.1 可自动执行
1. 新增证据锚点
2. 给已有节点补低风险关系
3. 更新用户状态
4. 新增局部理解层节点
5. 新节点初步归类
6. 新节点层级初判

### 14.2 建议自动执行但必须留痕
1. 基础关系初判
2. 初步抽象归位
3. 局部片区整理

### 14.3 必须走 proposal
1. 节点合并
2. 节点重命名
3. 抽象层升降
4. 主关系重构
5. 删除/归档节点
6. 去重执行

### 14.4 冻结原则

`LLM 可以自动生长地图，但不能静默改写地图骨架。`

## 15. ProposalCenter 的角色扩展

ProposalCenter 不再只是一处“压缩建议中心”，长期应扩展为：

`知识网络治理建议中心`

它至少承接两类 proposal：
1. `maintenance proposal`
2. `compression proposal`

其中本草案当前先冻结治理边界，不冻结完整页面设计。

## 16. 第一版冻结范围

第一版知识地图核心模型建议冻结为：
1. 混合节点体系
2. `foundation` 一等节点
3. 四层抽象结构
4. `KnowledgeNode + UserNodeState`
5. 五类公开关系
6. 证据附属层呈现
7. 用户侧 `FocusCluster`
8. 地图摘要页 -> 地图主视图
9. 工作地图视图作为默认主视图
10. 低成本模型的受约束解释链
11. LLM 自动增量维护 + proposal 治理边界

## 17. 当前明确不做

当前不冻结、不实现的内容包括：
1. 知识图高级编辑交互
2. 全局候选焦点簇系统
3. 复杂图布局算法
4. 批量 proposal 操作
5. 节点全量治理页面
6. 知识质量优化细则
7. 复杂聚类与大规模去重策略

原因：
1. 当前先冻结核心模型骨架，而不是一次性冻结全部产品能力。

## 18. 第一版核心对象清单

第一版建议强实现的核心对象只有 5 类：
1. `KnowledgeNode`
2. `EvidenceRef`
3. `UserNodeState`
4. `KnowledgeRelation`
5. `FocusCluster`

### 18.1 强实现对象说明

#### `KnowledgeNode`
必须先有。

原因：
1. 它是知识地图本体。
2. 没有稳定节点，后面的状态、关系、焦点簇都会漂。

#### `EvidenceRef`
必须尽早有。

原因：
1. 这个系统不是纯抽象图，必须可回溯。
2. 先把证据锚点挂法立住，节点才不会变成空抽象容器。

#### `UserNodeState`
必须有，但应晚于节点和证据。

原因：
1. 它表达“这个用户当前与节点是什么关系”。
2. 它不该反向污染知识本体定义。

#### `KnowledgeRelation`
必须有，但第一版只实现最小公开关系集。

原因：
1. 地图价值来自关系。
2. 但关系必须建立在稳定节点和证据边界之上。

#### `FocusCluster`
必须有，但它是视图级组织对象，不是知识本体。

原因：
1. 摘要页、主视图入口、预生成解释、用户 pin 都需要稳定宿主。
2. 如果没有独立对象，这些地方会各自重算并逐渐漂移。

### 18.2 第一版先不强实现的对象

这些对象长期可能存在，但第一版先不作为强落地点：
1. `GlobalClusterTemplate`
2. `GraphGovernanceRecord`
3. `NodeMergeExecution`
4. `GraphViewPreset`

说明：
1. 第一版先把用户侧知识地图入口跑通。
2. 治理系统、全局候选簇系统、复杂视图模板系统都先不进入主线。

## 19. 第一版对象落地顺序

第一版知识地图对象化落地顺序冻结为：
1. `KnowledgeNode`
2. `EvidenceRef`
3. `UserNodeState`
4. `KnowledgeRelation`
5. `FocusCluster`

### 19.1 顺序理由

#### 1. `KnowledgeNode` 先落
原因：
1. 它是知识地图本体。
2. 没有稳定节点，就没有后续对象的稳定宿主。

#### 2. `EvidenceRef` 第二个落
原因：
1. 先把证据锚点挂法立住，保证地图不是空抽象。
2. 这样后续节点和关系都能保持可回溯。

#### 3. `UserNodeState` 第三个落
原因：
1. 节点和证据稳定后，才适合叠加用户掌握状态。
2. 这样可以避免把用户状态误写进知识本体。

#### 4. `KnowledgeRelation` 第四个落
原因：
1. 关系虽然重要，但不能先于节点和证据边界稳定下来。
2. 第一版只做最小公开关系集，不扩复杂治理关系。

#### 5. `FocusCluster` 最后落
原因：
1. 它依赖节点、证据、状态和基础关系。
2. 它是入口组织对象，不应先于地图本体存在。

### 19.2 冻结规则

后续实现时不应：
1. 因为 `FocusCluster` 在 UI 上显眼，就把它提前到节点和证据之前实现。
2. 因为关系在理论上重要，就跳过节点与证据先做复杂关系层。
3. 把第一版直接扩成全局候选簇或复杂治理系统。

---

## 20. 一句话总结

第一版知识地图的本质不是“把项目经验画成图”，而是：

`以稳定知识节点为本体、以用户状态为投影、以项目事实为证据、以 FocusCluster 为入口、以 LLM 为低成本抽象维护器的长期个人知识地图工作台。`
